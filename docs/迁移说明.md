# Trend2SKU v1 到 v2 迁移说明

## 1. 迁移范围

v1 是固定兴趣消费命题输入：调用方主要提供 `brief`，候选名称、功能组合、用户镜像和商品风险大体沿固定演示路径生成。v2 将决策入口升级为九项结构化输入，并让同一条确定性工作流根据输入动态改变候选、验证内容、风险和报告上下文。

九项输入是 `brief`、`product_category`、`custom_category`、`target_segment`、`target_market`、`price_band`、`ip_strategy`、`objectives` 和 `constraints`。它们由同步 API、SSE、运行状态、结果视图和两类 Markdown 报告贯通，前端不再内置一条已填写的固定简报。

本次升级没有改变准确性边界：400 条评论仍是固定种子生成的合成离线演示数据；候选总分仍是规则量表而非爆款概率；模拟接受度和 NPS 仍不是真实用户研究；`GO` 仍只表示进入人工复核与下一验证阶段。

## 2. 决策语义变化

v2 的输入作用于以下环节：

- 候选生成：品类、客群、市场、价格带、IP 策略、目标和约束共同改变三条路径的候选名称、价值主张、功能与差异化表达。
- 用户镜像：访谈角色、异议、必改项和模拟接受度围绕当前候选与目标客群生成。
- 商品专家：成本与毛利、供应链、IP/合规、本地化风险围绕当前候选和约束评估。
- 报告：完整报告与开题摘要回显规范化后的本轮输入，避免把另一轮条件误写进报告。

九项输入**不直接向八维评分加分**。v2 不存在市场、IP 策略或目标选项到固定加分值的映射；输入可以先改变候选特征和风险，再间接改变确定性评分与排序。严重 IP 或质量风险仍可阻断直接 `GO`。

候选联动统一使用候选 ID。榜单选中项、商品概念、八维分数、模拟访谈/NPS 和商品风险必须引用同一个 ID，调用方不得继续依赖候选数组位置。

## 3. API 兼容变化

### 保持兼容

- `POST /api/run`、`GET /api/stream`、`POST /api/resume`、`GET /api/report` 和 `GET /api/health` 路径保持不变。
- v1 的 `category=interest_goods`、`brief`、`hitl` 和 `thread_id` 调用方式继续保留；未提供新增字段时使用 v2 的确定性默认值。
- SSE 事件顺序仍是若干 `trace`，随后一个 `result` 或 `error`，最后一个 `done`。
- 报告接口仍要求显式 `run_id`，`kind` 仍为 `full` 或 `opening`。
- HITL 的批准动作仍为 `{"thread_id":"...","action":"approve"}`，thread/checkpoint 冲突仍返回 `409`。

### 新增契约

- `POST /api/run` 接受九项结构化输入，并在响应的 `decision_input` 中回显规范化结果。
- `GET /api/stream` 接受对应 query；多选 `objectives` 使用重复 query 键，例如 `objectives=social&objectives=margin`。
- 同步结果和 SSE 的 `result.view` 均携带 `decision_input`。
- 健康检查同时返回 `configured_provider` 与 `effective_provider`，用于区分“配置为远程模型”和“本次实际以 offline 执行”。
- 浏览器报告入口改为页面内 `dialog`，同时提供 Markdown 报告和当前运行 JSON 下载；底层 `/api/report` 路径保持兼容。

### 破坏性变化

- 请求模型继续拒绝未知字段；拼写错误不会被静默忽略。
- `brief` 去除首尾空白后不能为空，最大 500 字符；`custom_category` 最大 40 字符；`constraints` 最大 300 字符。
- 枚举字段必须使用 v2 约定值；`product_category=other` 时 `custom_category` 必填，选择其他品类时该字段被规范化为空字符串。
- `objectives` 必须选择 1 至 4 项，重复值按首次出现顺序去重；旧客户端不能再把逗号分隔字符串当作数组发送。
- 候选 ID 仍稳定为三条生成路径的标识，但候选名称、功能、访谈、风险、分数、排序和榜首不再保证与固定 v1 演示相同。
- 前端运行前执行完整表单校验，简报初始为空；旧的自动化脚本若假设页面加载即包含固定简报，需要显式填写九项输入后再提交。
- 开始新一轮运行会清除旧报告链接和下载状态；外部脚本必须保存目标运行的 `run_id`，不能读取“最近一次”隐式状态。

## 4. 在线叙述迁移

v2 在线 Coding Plan 主路径统一为 Qwen：

```text
MINISO_LLM_PROVIDER=qwen
QWEN_API_KEY=<运行时安全注入>
QWEN_BASE_URL=https://coding.dashscope.aliyuncs.com/v1
QWEN_MODEL=qwen3.7-plus
QWEN_ENABLE_THINKING=false
```

协议为 OpenAI-compatible `POST /chat/completions`。Qwen 只负责叙述增强，不改候选的确定性数值、八维权重、阈值或严重风险规则。

MiniMax provider 与现有 `MINIMAX_*` 变量保留用于旧调用方兼容，但不是 v2 新接入的主推配置。需要从 MiniMax 迁移时，只替换 provider 与对应的 Qwen 配置，不修改业务请求模型、评分规则或 API 路径。

## 5. 安全与显式降级

- 不把 Key 写入源码、请求示例、trace、报告、截图、ZIP 或命令历史；可通过获授权的运行环境注入，或仅保存在 Git 忽略、权限为 `0600`、绝不打包的本地 `.env`。
- Qwen 客户端只记录脱敏错误类别、安全 request ID 和 token 计数，不回显完整提示词、Key 或上游响应正文。
- 配置 `MINISO_LLM_PROVIDER=qwen` 但 Key 缺失时，服务启动后明确报告 `configured_provider=qwen`、`effective_provider=offline`。
- 首个远程请求发生超时、HTTP 错误、解析错误或响应结构错误时，网关显式熔断到 `offline`；后续叙述使用确定性实现，流程不会把远程失败伪装成在线成功。
- 降级只改变叙述来源，不改变数值计算与风险闸口。在线与离线比较时必须同时记录 configured/effective provider，不能仅按启动参数归因。
- 九项输入被视为不可信用户输入：API 执行长度、枚举和组合校验；前端使用文本节点渲染，不将输入或模型文本拼接为可执行 HTML。
- 报告读取必须绑定明确 `run_id`；SSE 客户端必须校验每个事件的 `run_id` 与 `thread_id`，防止并发结果串线。
- 打包器使用正向 allowlist，拒绝真实环境文件、密钥痕迹、本机用户目录、缓存、运行记录、旧截图/报告及 `docs/superpowers`；图片等二进制资产不按文本误扫。

## 6. 测试与验收

迁移后至少执行以下本地验证，结果以命令实际输出为准：

```bash
PYTHONPATH=backend .venv/bin/python -m pytest backend/tests -q
node --check frontend/app.js
.venv/bin/python -m compileall -q backend scripts run.py
.venv/bin/python scripts/pre_pr_check.py
```

重点回归范围：

- 九项输入的规范化、默认值、枚举、长度、自定义品类与目标数量边界。
- 两组不同输入确实生成不同候选、用户验证、商品风险和报告，同时保持三条候选 ID 独立。
- 同步 API 与 SSE 完整传播 `decision_input`，重复 `objectives` query 不丢项。
- 候选切换联动概念、得分、验证和风险；报告 `dialog`、Markdown 下载与 JSON 下载绑定当前 `run_id`。
- Qwen 的 OpenAI-compatible 请求、关闭 thinking、元数据解析、错误脱敏、缺 Key 与首错熔断。
- 四个浏览器视口的文本容纳、无页面级溢出、恶意文本转义和真实操作状态。
- 打包后的 allowlist、MANIFEST、ZIP CRC、外部 SHA-256、隐私扫描，以及解压后全量测试。

完整的逐项执行顺序见 `deliverables/提交清单_v2.md`。清单默认全部未勾选，不能用文档描述替代真实验证，也不能虚构在线 smoke 已成功。
